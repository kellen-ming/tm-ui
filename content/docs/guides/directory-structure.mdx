## 推荐理解方式

先把目录分成五层：

- 生成层
- 路由层
- 内容层
- 示例与文档组件层
- source 层

## 每个目录分别负责什么

<Accordions type="multiple">
  <Accordion title=".source">
    这是 Fumadocs 在 dev 或 build 过程中自动生成的中间层。它会把 `source.config.ts` 里的规则和 `content/docs` 里的文档扫描结果整理成可直接 import 的映射文件。一般不手改，把它理解成“编译产物”最合适。
  </Accordion>
  <Accordion title="app/docs">
    负责文档站的路由层。`layout.tsx` 是 docs 外壳，`[[...slug]]/page.tsx` 是具体页面渲染器。
  </Accordion>
  <Accordion title="app/layout.tsx 与 app/page.tsx">
    这是整个站点的根布局和首页。`app/layout.tsx` 负责全局 provider 与基础样式，`app/page.tsx` 还是当前项目原有的 showcase 入口，不属于 Fumadocs 内容层。
  </Accordion>
  <Accordion title="content/docs">
    负责真正的文档内容。以后新增页面，优先在这里写 MDX，而不是先写 React 页面。
  </Accordion>
  <Accordion title="content/docs/*/meta.json">
    `meta.json` 不是正文内容，它更像“当前这一层的导航说明书”。它主要负责两件事：定义这一层 section 的展示标题，以及控制当前层直接孩子节点的顺序，所以它会直接影响左侧导航的 page tree。
  </Accordion>
  <Accordion title="content/docs/*.mdx">
    `.mdx` 是真正的文档正文。它既能写 Markdown，也能直接写 React 组件标签，所以特别适合写组件文档、说明文档和带示例的页面。
  </Accordion>
  <Accordion title="mdx-components.tsx">
    这是 MDX 组件映射表。它负责告诉系统：当文档里写 `<Tabs />`、`<Files />`、`<ButtonBasicExampleShowcase />` 时，实际要渲染哪个 React 组件。没有这个映射，MDX 只认识普通 Markdown，不认识这些自定义标签。
  </Accordion>
  <Accordion title="components/docs">
    这里放文档专用的可复用组件，比如 `ExampleShowcase` 这种“上面 preview、下面 code”的展示容器。它们不是业务 UI 组件，而是帮助文档组织内容的 docs helper。
  </Accordion>
  <Accordion title="examples">
    放文档里的示例代码和示例组件。它们不是通用 UI 组件，而是帮助读者理解组件用法的 example。通常一个 example 会同时包含预览实现、展示代码字符串，以及一个对文档友好的包装入口。
  </Accordion>
  <Accordion title="source.config.ts">
    这是 Fumadocs 的内容源配置入口。这里告诉系统文档目录在哪、文档集合怎么定义、MDX 处理规则是什么。你可以把它理解成“扫描和处理文档的规则定义处”。
  </Accordion>
  <Accordion title="lib/source.ts">
    这是站点运行时真正消费的 source 入口。它会 import `.source/server` 里自动生成的 `docs`，再通过 `docs.toFumadocsSource()` 和 `loader()` 把原始文档集合包装成 `getPage()`、`getPages()`、`pageTree`、`generateParams()` 这些能力。
  </Accordion>
</Accordions>

## 什么时候需要新建自定义组件

当出现这些情况时，建议从 MDX 抽成单独的 example 或 docs 组件：

- 页面里有复杂交互
- 同一段内容会被多个页面复用
- 需要和现有业务组件联动
- 纯 MDX 已经让页面变得难维护

## 一个实用判断标准

- 如果它是“帮助文档排版和组织内容”的，优先放 `components/docs`
- 如果它是“演示某个组件如何使用”的，优先放 `examples`
- 如果它是“真正可复用的业务或设计系统组件”，继续放 `components/ui` 或 `components/feature`
