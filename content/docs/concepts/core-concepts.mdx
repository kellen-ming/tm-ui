## 1. Content Layer

Fumadocs 的第一层是文档内容本身，也就是：

- `meta.json`
- `.mdx`
- 目录结构

它的核心价值是把“文档内容”从“页面实现”里拆出来，让组件库站点可以边长内容、边迭代 UI。

## 2. Source Layer

在当前项目里，`source` 是文档内容和 Next.js 路由之间的桥。

它负责：

- 根据 slug 找到页面
- 生成侧边栏 page tree
- 为 SSG 生成静态参数
- 暴露页面 metadata 所需的数据

## 3. Layout vs Page

这一点非常重要：

- `layout` 决定 docs 的外壳、导航、侧边栏、主题等公共结构
- `page` 决定当前 MDX 页面如何显示标题、描述、正文和 TOC

当我们以后继续加搜索、加 AI、加组件导航时，优先动 `layout`；
当我们优化某一篇文档页的阅读体验时，优先动 `page` 或 MDX 组件。

## 4. MDX Components

MDX 页面里能直接写 `<ButtonDemo />`、`<Files />`、`<Tabs />`，本质上都依赖统一的组件映射。

在这个项目里，它的职责是：

- 暴露 `fumadocs-ui` 的现成组件
- 暴露自定义 docs 组件
- 保持文档写作体验统一

## 常见误区

<Accordions type="single" collapsible>
  <Accordion title="Fumadocs 会替代现有展示逻辑吗？">
    不会。它更适合作为文档框架层，而不是替代你已有的组件源码、交互 demo 和 props 配置体系。
  </Accordion>
  <Accordion title="是不是必须先做 Chatbot？">
    不是。更合理的顺序是先把内容层和搜索做稳，再让 AI 基于真实文档和组件元数据工作。
  </Accordion>
  <Accordion title="MDX 页面能不能直接塞很多业务逻辑？">
    可以，但不建议。复杂交互更适合抽到自定义组件里，再通过 MDX 组合使用。
  </Accordion>
</Accordions>
